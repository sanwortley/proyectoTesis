1. Visión General del Sistema

1.1 ¿Qué es Pro Cup Padel?

Pro Cup Padel es una aplicación web full-stack para la gestión integral de torneos de pádel en modalidad dobles/parejas. Cubre el ciclo completo: inscripción de jugadores y equipos, generación de grupos y fixture, registro de resultados por sets, bracket de playoff automático, ranking acumulado histórico y notificaciones por email.

Problema que resuelve: Los torneos de pádel se gestionan manualmente con planillas y WhatsApp. Pro Cup Padel automatiza todos estos procesos en una sola plataforma web accesible desde cualquier dispositivo.

3. Módulos del Backend

📁 backend/src/ — Node.js + Express 5.1 (ESM)
Entry Pointapp.mjs
ExpressCORSdotenvnode-cronStatic files

Punto de entrada de toda la aplicación backend. Hace tres cosas principales:

  1. Configura Express: activa CORS (permite solicitudes desde cualquier origen), parseo de JSON y sirve la carpeta uploads/ como archivos estáticos para que el frontend acceda a las fotos de perfil con /uploads/perfiles/foto.jpg.
  2. Monta todas las rutas bajo el prefijo /api: playoff, rutas principales, auditoría, auth, ranking, torneos y dashboard.
  3. Programa el cron job con node-cron (expresión '0 * * * *' = cada hora en el minuto 0). El job busca torneos cuya fecha_inicio esté entre 23 y 25 horas en el futuro y envía emails de recordatorio a todos los jugadores inscriptos. Un Set en memoria evita enviar el mismo email dos veces en la misma sesión del proceso.
Configconfig/db.js
PostgreSQLPool de conexiones

Crea y exporta un pool de conexiones a PostgreSQL usando la librería pg. Un pool reutiliza conexiones abiertas en lugar de abrir una nueva por cada query, lo que mejora el rendimiento significativamente. Las credenciales se leen de variables de entorno: DB_HOST, DB_PORT, DB_USER, DB_PASSWORD, DB_NAME. En producción (Railway), estas las provee automáticamente la variable DATABASE_URL.

Todas las rutas importan este pool y lo usan con pool.query() para consultas simples o pool.connect() para obtener un cliente que permite transacciones (BEGIN / COMMIT / ROLLBACK).

Configconfig/multer.js
MulterUpload de archivosValidación MIME

Configura Multer para recibir fotos de perfil. Define tres aspectos clave:

  • Destino: guarda los archivos en uploads/perfiles/ (la crea si no existe).
  • Nombre único: genera el nombre con timestamp + número aleatorio para evitar colisiones entre archivos (perfil-1720000000-123456789.jpg).
  • Filtro de tipo: solo acepta JPEG, JPG, PNG y WebP (valida tanto extensión como MIME type). Si el archivo no pasa, rechaza con error. Límite de 5 MB por archivo.

Rutaroutes/auth.js — Middleware JWT
JWTMiddlewareAuthorization header

Exporta el middleware requireAuth que protege las rutas sensibles. Extrae el token del header Authorization: Bearer <token>, lo verifica con jwt.verify() y, si es válido, adjunta req.user = { id, role } para que el siguiente handler sepa quién es el usuario. Si el token es inválido o expiró (6h), devuelve 401.

export function requireAuth(req, res, next) {
  const token = req.headers.authorization?.slice(7); // "Bearer xxx"
  const payload = jwt.verify(token, process.env.JWT_SECRET);
  req.user = { id: payload.id, role: payload.role };
  next(); // continúa al siguiente middleware/handler
}
Controllercontrollers/authController.js
Login alternativoDiagnósticoIP realbcrypt dual

Controlador de login alternativo (montado a través de auth.js). A diferencia del login en routes/index.js, este controlador incluye herramientas de diagnóstico:

  • IP real: lee el header x-forwarded-for para obtener la IP real del cliente detrás de proxies (Railway usa proxies).

Exporta generarGruposAleatorios(equipos). Primero mezcla aleatoriamente los equipos (Math.random() - 0.5) y luego los distribuye según la cantidad total:

La función cortarParejo(lista, n) distribuye los equipos de a uno por grupo en modo rotativo, garantizando que la diferencia entre grupos sea como máximo 1 equipo.

Utilutils/generarPlayoff.js
AlgoritmoSnake-seedingBYEnext_match_id

Implementa generarPlayoffSiNoExiste(torneoId). Algoritmo en 6 pasos:

  1. Extrae top 2 de cada grupo ordenado por puntos → sets a favor → games a favor.
  2. Snake-seeding: 1ro Grupo A vs 2do Grupo B, 1ro Grupo B vs 2do Grupo A, etc. Evita que los mejores se enfrenten antes de la final.
  3. Determina rondas por potencia de 2: 4 equipos → SEMIS+FINAL, 8 → CUARTOS+SEMIS+FINAL, 16 → OCTAVOS+…
  4. Asigna BYE a los mejores seeds si el número no es potencia exacta de 2.
  5. Casos especiales: 3 grupos (6 equipos) → hardcoded con play-in. 1 grupo → va directo a final/semis/cuartos.
  6. Encadena los partidos: guarda next_match_id y next_slot para que el auto-avance funcione sin intervención.
Utilutils/mailer.js
Resend SDKEmail HTMLLazy initResiliente

Encapsula el envío de emails con Resend SDK. El cliente se inicializa de forma lazy: solo si RESEND_API_KEY existe en el entorno, evitando crash en desarrollo sin credenciales. Si la variable no existe, las funciones retornan silenciosamente sin error.

const resend = process.env.RESEND_API_KEY
  ? new Resend(process.env.RESEND_API_KEY)
  : null; // sin clave → no crash, solo no envía

Exporta dos funciones: enviarConfirmacionInscripcion() (al inscribirse) y enviarRecordatorioTorneo() (invocado por el cron job). Ambos usan HTML inline con el estilo de Pro Cup Padel.

Utilutils/logIngreso.js + utils/fileAudit.js
Auditoría dualCSV + JSONLPromise.allSettled

logIngreso.js es el punto central de auditoría. Se llama desde cada resultado posible del login (éxito, usuario no encontrado, contraseña incorrecta, error interno). Escribe en dos destinos independientes:

  1. Tabla audit_log_ingresos en PostgreSQL (dentro de try/catch propio).
  2. Archivos de log vía fileAudit.js.

fileAudit.js mantiene dos archivos en ./logs/: un CSV y un JSONL (una línea JSON por evento). Usa Promise.allSettled() para escribir ambos en paralelo sin que el fallo de uno afecte al otro. Si el directorio no existe, lo crea automáticamente.

5. Flujo Completo de un Torneo

1
[Organizador] Crea el torneo — Completa el formulario en /crear-torneo: nombre, fechas, formato (categoría fija o suma), modalidad (fin de semana o liga) y máximo de equipos. El backend valida y guarda con POST /api/torneos.
2
[Jugadores] Se registran e inscriben — Cada jugador se registra en /registro. Luego va a /inscripcion, elige el torneo y su compañero. El backend valida categorías, fechas superpuestas y envía emails de confirmación a ambos.
3
[Organizador] Genera grupos — Desde /organizador/torneos genera los grupos. El sistema distribuye aleatoriamente los equipos y crea el fixture. Para liga, calcula las fechas de cada jornada según los días configurados.
4
[Organizador] Carga resultados de grupos — Desde /cargar-resultado ingresa los sets de cada partido. El sistema actualiza la tabla de posiciones. Cuando se carga el último partido pendiente → el playoff se genera automáticamente.
5
[Organizador] Carga resultados del playoff — Carga resultado partido por partido. El ganador avanza automáticamente al siguiente encuentro. Al terminar la FINAL → el sistema genera el ranking del torneo automáticamente.
6
[Todos] Visualizan resultados y ranking — Cualquier usuario (incluso invitados) puede ver el bracket en /torneosllave y el ranking acumulado en /ranking, filtrado por categoría.

8. Preguntas Frecuentes de Defensa

¿Por qué Node.js y no otro lenguaje backend?

JavaScript en el backend permite compartir lógica con el frontend y es ideal para APIs REST por su modelo de I/O no bloqueante (event loop), que maneja eficientemente múltiples conexiones simultáneas sin threads.

¿Por qué PostgreSQL y no MongoDB?

Los datos son fuertemente relacionales: jugadores → equipos → inscripciones → grupos → partidos. El modelo relacional con FK y transacciones garantiza integridad, especialmente en operaciones como inscripción donde varios inserts deben ser atómicos (todos o ninguno).

¿Cómo funciona JWT?

Token con tres partes en base64: header (algoritmo), payload (id y rol del usuario) y firma (HMAC-SHA256 con JWT_SECRET). El backend verifica la firma en cada request sin consultar la DB, lo que hace la autenticación muy eficiente y sin estado.

¿Cómo se evita que dos jugadores se inscriban en torneos superpuestos?

La ruta POST /api/inscripcion hace una query SQL que busca si alguno de los dos jugadores tiene inscripciones en otros torneos cuyos rangos de fechas se solapan con el torneo nuevo: NOT (fecha_fin < fecha_inicio_nuevo OR fecha_inicio > fecha_fin_nuevo).

¿Qué es Docker Compose y por qué se usa?

Permite definir en un YAML los 3 servicios (DB, backend, frontend) con sus configs y redes. Con docker-compose up --build se levanta todo el sistema en cualquier máquina sin instalar dependencias manualmente, garantizando reproducibilidad.

¿Por qué valor_numerico y no id_categoria en el ranking?

Porque los torneos de tipo "suma" no tienen categoría fija: dos jugadores de distinta categoría juegan juntos (ej: 5ta+4ta=SUMA9). El valor_numerico individual es el único denominador común para agrupar el ranking coherentemente entre ambos tipos de torneo.

¿Cómo funciona el auto-avance en el playoff?

Al crear el bracket, cada partido de playoff tiene next_match_id (FK al partido siguiente) y next_slot (1 o 2). Al cargar el resultado, el sistema calcula el ganador y ejecuta: UPDATE partidos_llave SET equipo{slot}_id = ganador_id WHERE id = next_match_id.

¿Por qué React Context y no Redux?

El estado compartido del sistema es simple: solo el usuario autenticado. Context es suficiente y evita la complejidad de Redux (actions, reducers, store). Redux se justificaría en apps con múltiples fuentes de estado complejo y sincronización frecuente.

¿Qué pasa si el servidor de email no está configurado?

El sistema es resiliente: el cliente Resend se inicializa solo si RESEND_API_KEY está definida. Si no, las funciones retornan silenciosamente sin error. El flujo de inscripción y login completa igual.
