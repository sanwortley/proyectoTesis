Requerimientos del Sistema

Requerimientos Funcionales

  • Gestión de Torneos: El sistema permite crear torneos configurando nombre, formato (Categoría Fija o Suma de categorías), modalidad (Fin de semana o Liga), fechas, límite de equipos y, si corresponde, los días de juego semanales.
  • Inscripción de Equipos: El organizador puede inscribir parejas de jugadores ya registrados en el sistema, generando automáticamente el nombre del equipo (Apellido1/Apellido2). El sistema valida compatibilidad de categoría según el formato del torneo.
  • Generación de Grupos: Al cerrar las inscripciones el sistema distribuye los equipos en grupos de forma automática, usando snake-seeding para una distribución equitativa.
  • Carga de Resultados — Fase de Grupos: El organizador ingresa los resultados de cada partido set a set (games por equipo en cada set). El sistema calcula el ganador, actualiza puntos, sets y games de cada equipo en su grupo.
  • Carga de Resultados — Play-off: El organizador carga resultados de los partidos de la llave eliminatoria. El sistema determina el ganador y avanza al equipo a la siguiente ronda.
  • Generación Automática de Play-off: Al completarse todos los partidos de grupos el sistema genera automáticamente el bracket eliminatorio usando snake-seeding entre los clasificados (primeros, segundos y, en ciertos casos, el mejor tercero).
  • Ranking de Jugadores: El sistema calcula y muestra un ranking global de jugadores con puntos por fase alcanzada en torneos, incluyendo bonus por partidos ganados, sets y efectividad en grupos.
  • Autenticación y Roles: El sistema gestiona usuarios con roles diferenciados (organizador / jugador). El acceso a funciones administrativas está restringido al rol organizador mediante JWT con expiración de 6 horas.
  • Registro de Ingresos: El sistema registra en un log de auditoría cada inicio de sesión exitoso, guardando usuario, rol, IP y timestamp.
  • Filtrado de Partidos: En la vista de carga de resultados el organizador puede buscar partidos por nombre de equipo o jugador. En modalidad liga puede filtrar por fecha de jornada.

Requerimientos No Funcionales

  • Seguridad: Contraseñas almacenadas con hash bcrypt (factor 10). Rutas protegidas con middleware JWT. Validaciones en backend contra inyección SQL mediante consultas parametrizadas.
  • Disponibilidad: El sistema se despliega mediante Docker Compose (backend, frontend, PostgreSQL) garantizando un entorno reproducible.
  • Rendimiento: Las consultas de ranking y generación de play-off se ejecutan en tiempo inferior a 2 segundos para torneos de hasta 64 equipos.
  • Usabilidad: Interfaz web responsiva desarrollada en React 19 con navegación por roles y retroalimentación visual inmediata ante acciones del usuario.
  • Mantenibilidad: Backend estructurado en Express 5.1 con rutas separadas por módulo funcional. Frontend con componentes reutilizables y hooks personalizados.

Diagrama de Asociación — Iniciar Sesión

Fig. 6 — Diagrama de casos de uso para el módulo Iniciar Sesión

Caso de Uso: Iniciar Sesión
Actor principal: Usuario (organizador o jugador)
Precondición: El usuario está registrado en el sistema
Postcondición: El usuario obtiene un JWT válido (6 h) y el acceso queda registrado en el log de auditoría
Curso Normal: Curso Alternativo:
  1. El actor accede a la pantalla de inicio de sesión.
  2. Ingresa su nombre de usuario (insensible a mayúsculas).
  3. Ingresa su contraseña.
  4. Presiona "Ingresar".
  5. El sistema busca al usuario en la BD de forma insensible a mayúsculas (ILIKE).
  6. El sistema verifica la contraseña usando bcrypt.compare().
  7. El sistema genera un JWT firmado con expiración de 6 horas.
  8. El sistema registra el ingreso en audit_log_ingresos (usuario, rol, IP, timestamp).
  9. El sistema redirige al usuario a la vista correspondiente a su rol.
  10. Fin del caso de uso.
  5. 5.1 Si el usuario no existe, el sistema devuelve "Usuario o contraseña incorrectos" (mensaje genérico por seguridad).
  6. 6.1 Si la contraseña no coincide con el hash almacenado, el sistema devuelve el mismo mensaje genérico.
  8. 8.1 Si falla el registro en el log, se continúa sin interrumpir el flujo del usuario.
  9. 9.1 Si el JWT está expirado en futuras solicitudes, el sistema devuelve 401 y redirige al login.

Fig. 12 — Diagrama de casos de uso para el módulo Inscripción de Equipo

Caso de Uso: Inscripción de Equipo
Actor principal: Organizador
Precondición: El torneo está en estado "abierto" y no alcanzó el límite de equipos. Ambos jugadores están registrados en el sistema.
Postcondición: El equipo queda inscripto en el torneo con nombre auto-generado. El cupo disponible se reduce en uno.
Curso Normal: Curso Alternativo:
  1. El organizador accede al torneo y selecciona "Inscribir Equipo".
  2. El sistema muestra el formulario con los desplegables de jugadores disponibles, mostrando nombre, apellido y categoría de cada uno.
  3. El organizador selecciona el Jugador 1 del desplegable.
  4. El organizador selecciona el Jugador 2 del desplegable.
  5. El sistema valida la compatibilidad de categoría: si el torneo es "Categoría Fija", ambos jugadores deben tener la misma categoría que el torneo; si es "Suma", la suma de las categorías debe coincidir con la suma objetivo.
  6. El sistema genera automáticamente el nombre del equipo (Apellido1/Apellido2).
  7. El organizador presiona "Inscribir Equipo".
  8. El sistema guarda el equipo en la tabla equipos y confirma la inscripción.
  2. 2.1 Si no hay jugadores disponibles, el sistema muestra "No hay jugadores registrados".
  4. 4.1 Si el jugador 2 es el mismo que el jugador 1, el sistema muestra un error de selección.
  5. 5.1 Si las categorías no son compatibles, el sistema muestra "Categoría incompatible con el torneo" y deshabilita el botón de inscripción.
  8. 8.1 Si el torneo ya alcanzó el límite de equipos, el sistema informa que el torneo está completo.
Caso de Uso: Inscripción de Equipo (continuación)
Curso Normal: Curso Alternativo:
  1. El sistema verifica que ninguno de los dos jugadores esté ya inscripto en el mismo torneo con otro equipo.
  2. El sistema verifica que no exista solapamiento de fechas con otros torneos en los que los jugadores participen.
  3. El equipo queda visible en la lista de inscriptos del torneo.
  4. Fin del caso de uso.
  1. 9.1 Si algún jugador ya está inscripto en el torneo, el sistema muestra "El jugador ya está inscripto en este torneo".
  2. 10.1 Si se detecta solapamiento de fechas con otro torneo, el sistema avisa y solicita confirmar.
